# Case Study Page Pattern

Shipped 2026-04-26. The first instance is `/case-study/spy-2026-2007`,
which tells the "SPY today rhymes with autumn 2007" story end-to-end.

## What it is

A scroll-driven, single-slug page that does three things at once:

1. Uses real engine data (real SPY closes from the canonical parquet).
2. Acts as a scrolly explainer of the platform.
3. Tells a side-by-side "this rhymes with that" story with a forecast
   reveal.

It is NOT the [[demo]] page (a video trailer of the workstation), and
it is NOT the [[prudent-demo]] (a longer interactive showcase of the
NL-to-pattern surface). The case study is a story-shaped surface
focused on a specific historical analog — a single concrete example
that an investor or finance-Twitter reader can verify themselves.

## When to add a new one

Add a case study when there is a specific historical pattern worth
telling a story about — a regime change, a top, a bottom, a specific
analog the engine surfaces consistently. Each case study is its own
hardcoded slug under `app/case-study/<slug>/page.tsx`. We do NOT use
a `[slug]` dynamic route until there are 3+ case studies; one slug
is simpler to ship and tune.

Don't write a case study for ephemeral events. The page is meant to
remain on the site for months as a permanent worked example, not a
weekly market commentary.

## File layout

```
the-similarity-app/
├── app/case-study/spy-2026-2007/
│   ├── page.tsx          # the route ("use client", scroll IO, 5 sections)
│   └── data.ts           # generated by build-case-study-data.py
├── components/case-study/
│   └── rhyme-chart.tsx   # purpose-built SVG chart for this surface
└── scripts/
    └── build-case-study-data.py  # parquet → static TS module
```

## Data flow

Real parquet (`the-similarity-data/data/stocks/spy/1d.parquet`) →
Python generator (`scripts/build-case-study-data.py`) → static TS
module (`app/case-study/<slug>/data.ts`) → page.

The generator is committed AND its output is committed. The page
builds without any data-package dependency at runtime, which keeps
the route statically prerenderable and fast on first paint. Re-run
the generator whenever the parquet refreshes (the daily data cadence
workflow could trigger this; for now it's manual).

Why static instead of live `searchApi`: a cold backend call would
blow past the 200ms first-paint budget that this surface needs. A
later iteration can swap in a live engine call at build-time
(`generateStaticParams` + a build-time fetch) once the registry
exposes a stable analog-of-record per slug.

## Five-section layout

1. **Hook** — full-viewport headline + sub-line + scroll hint.
2. **Setup** — the present chart alone.
3. **Match** — present + analog side-by-side with score strip.
4. **Reveal** — analog continuation draws past the match end; a
   forecast cone emerges from the present line tip on the left.
5. **Verify + close** — workstation deep-link CTA + 4-paragraph
   platform explainer + signoff.

## Reveal mechanics

Vanilla IntersectionObserver only — no scrollytelling library. Each
`<section>` carries a ref + a `useInView` hook flipping `data-state`
between "enter" and "active". CSS owns the actual transitions. See
`useInView` in `app/case-study/spy-2026-2007/page.tsx` for the
threshold tuning rationale.

The reveal is sticky: once a section enters, it stays "active". This
matches what the audience expects from a case study (re-scrolling
shouldn't replay every animation).

## Workstation deep-link contract

The "Open in the workstation" CTA encodes its query via
[[lib/url-state]]'s `serializeUrlState` so the URL contract is
shared with all other surfaces that link into the workstation. The
encoded fields are minimal: dataset + queryLen + horizon + chartMode
+ surface. The workstation's parser is defensive, so missing fields
fall back to its own defaults.

## Voice

Builder-talking-to-builder, never marketing-deck. No em dashes, no
AI vocabulary. The chart does the work; copy is supporting voice.
See `the-similarity-app/app/case-study/spy-2026-2007/page.tsx` for
the locked copy register.
